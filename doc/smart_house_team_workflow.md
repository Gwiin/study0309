# 스마트 환경제어 하우스 협업 가이드

## 1. 문서 목적

이 문서는 `GitHub`에 익숙하지 않은 팀원이 각자 로컬 환경에서 개발하고, 각 보드에서 단독 테스트를 마친 뒤, 팀 프로젝트 저장소에 안전하게 업로드하여 최종 통합할 수 있도록 작업 순서를 정리한 협업 기준 문서이다.

이 문서를 먼저 읽고, 이후 각자 맡은 노드 문서를 따라 개발한다.

---

## 2. 전체 개발 방식

우리 팀은 아래 순서로 진행한다.

1. `공통 규격`을 먼저 확정한다.
2. 각 팀원은 자신의 `노드 전용 Pico 2 W`에서 단독 개발을 진행한다.
3. 각 노드는 `로컬 단독 테스트`를 완료한 뒤에만 GitHub에 업로드한다.
4. GitHub에서는 각자 `개인 브랜치`에서 작업하고, 검증 후 `main`에 합친다.
5. 마지막에 `MQTT 토픽`, `메시지 형식`, `임계값`, `상태 이름`을 맞춰 통합 테스트를 진행한다.

핵심 원칙은 `개별 완성 -> 규격 확인 -> 통합`이다.

---

## 3. 폴더 및 담당 권장 구조

권장 담당은 아래처럼 고정한다.

- `노드 A 담당`: 환경 센서 측정, 센서 데이터 가공, 환경 데이터 발행
- `노드 B 담당`: LED/서보 제어, 명령 수신 처리, 장치 상태 회신
- `노드 C 담당`: 전체 상태 저장, AUTO/MANUAL 관리, 제어 판단, 통합 테스트 주도

권장 브랜치 이름:

- `feature/node-a-sensor`
- `feature/node-b-actuator`
- `feature/node-c-controller`
- `integration/final-test`

---

## 4. GitHub 작업 순서

### 4-1. 처음 시작할 때

각 팀원은 처음에 아래 순서로 시작한다.

1. 팀 저장소를 `clone` 한다.
2. 최신 `main` 브랜치 상태를 확인한다.
3. 자신의 작업 브랜치를 만든다.
4. 자신의 노드 문서에 맞춰 개발한다.

예시 명령:

```bash
git clone <팀-저장소-주소>
cd study
git checkout main
git pull origin main
git checkout -b feature/node-a-sensor
```

### 4-2. 작업 중 반복 순서

작업 중에는 아래 흐름을 반복한다.

1. 로컬에서 코드 수정
2. 빌드 확인
3. 보드에 업로드 후 하드웨어 테스트
4. 변경 파일 확인
5. 커밋
6. 원격 브랜치에 push

예시 명령:

```bash
git status
git add <수정한 파일>
git commit -m "feat: add node A sensor read logic"
git push origin feature/node-a-sensor
```

### 4-3. main에 합치기 전

`main`에 바로 작업하지 않는다. 반드시 아래를 지킨다.

1. 자기 브랜치에서만 작업한다.
2. 자기 보드에서 기능 테스트를 끝낸다.
3. 문서의 `업로드 전 체크리스트`를 모두 통과한다.
4. 통합 담당 또는 팀원 전체가 메시지 규격을 다시 확인한다.
5. 그 뒤에만 `pull request` 또는 합의 후 merge 한다.

### 4-4. merge 직전 최신 코드 반영

다른 팀원이 먼저 코드를 올렸을 수 있으므로, merge 또는 pull request 전에 최신 `main`을 반영한다.

예시 명령:

```bash
git checkout main
git pull origin main
git checkout feature/node-a-sensor
git merge main
```

충돌이 나면 바로 억지로 해결하지 말고, 어떤 파일에서 충돌이 났는지 팀원과 먼저 확인한다.
특히 같은 `MQTT 토픽`, `공통 헤더`, `메시지 파서`를 여러 명이 동시에 건드렸다면 합의 후 하나의 형식으로 다시 맞춘다.

## 5. 팀 공통 합의 항목

통합을 쉽게 하려면 아래 항목을 반드시 고정한다.

### 5-1. 노드 역할 이름

- `NODE_A`: 환경 센서 노드
- `NODE_B`: 장치 제어 노드
- `NODE_C`: 중앙 관리 노드

### 5-2. 모드 이름

- `AUTO`
- `MANUAL`

### 5-3. 제어 명령 이름

- `LIGHT_ON`
- `LIGHT_OFF`
- `WINDOW_OPEN`
- `WINDOW_CLOSE`
- `MODE_AUTO`
- `MODE_MANUAL`

### 5-4. 장치 상태 이름

- `ON`
- `OFF`
- `OPEN`
- `CLOSE`

### 5-5. 센서 데이터 키 이름

- `light`
- `temp`
- `humidity`

`humidity`는 줄여 쓰지 않고 통일한다.

### 5-6. 권장 MQTT 토픽

- `house/env`
- `house/mode`
- `house/cmd/light`
- `house/cmd/window`
- `house/status/nodeA`
- `house/status/nodeB`
- `house/status/nodeC`
- `house/heartbeat/nodeA`
- `house/heartbeat/nodeB`
- `house/heartbeat/nodeC`

### 5-7. 메시지 형식 합의

센서 데이터는 한 가지 형식으로 고정한다.

```txt
light=275,temp=29.1,humidity=72.0
```

조명 명령:

```txt
ON
OFF
```

창문 명령:

```txt
OPEN
CLOSE
```

모드 명령:

```txt
AUTO
MANUAL
```

상태 보고 예시:

```txt
mode=AUTO,light=275,temp=29.1,humidity=72.0,lamp=ON,window=OPEN
```

### 5-8. 임계값 합의

처음 버전은 아래 값으로 고정하고, 나중에 필요 시 조정한다.

- 조명 ON 기준: `light < 280`
- 조명 OFF 기준: `light > 320`
- 창문 OPEN 기준: `temp > 28` 또는 `humidity > 70`
- 창문 CLOSE 기준: `temp <= 27` 그리고 `humidity <= 65`

### 5-9. 주기 합의

- 조도 측정 주기: `0.5초 ~ 1초`
- 온습도 측정 주기: `1초 ~ 2초`
- 환경 데이터 발행 주기: `1초`
- heartbeat 주기: `2초`
- 상태 출력 주기: `1초`

### 5-10. 파일 수정 범위 합의

통합 충돌을 줄이기 위해 처음부터 수정 범위를 나눈다.

- `노드 A 담당`: 센서 읽기 코드, 센서 관련 설정, 환경 데이터 발행 코드
- `노드 B 담당`: LED/서보 제어 코드, 장치 상태 보고 코드
- `노드 C 담당`: 상태 판단 코드, 모드 관리 코드, 명령 발행 코드
- `공통 파일`: 토픽 정의, 공통 상수, 메시지 형식 문서는 팀 합의 후 수정

같은 파일을 두 명 이상이 동시에 수정해야 할 경우에는 먼저 담당자를 정하고 순서대로 반영한다.

## 6. 각 팀원이 꼭 지켜야 할 개발 규칙

### 6-1. 독립 실행 가능

자기 노드는 다른 노드가 없어도 최소한 아래가 가능해야 한다.

- 전원 인가 후 부팅 확인
- `UART` 로그 출력
- 핵심 기능 단독 테스트
- 에러 발생 시 에러 로그 확인

### 6-2. 로그 형식 통일

UART 출력은 사람이 읽기 쉽게 통일한다.

예시:

```txt
[NODE_A] WIFI OK
[NODE_A] light=281 temp=27.8 humidity=66.0
[NODE_B] CMD LIGHT ON
[NODE_C] MODE AUTO
```

### 6-3. 하드코딩 값 표시

임계값, 핀 번호, Wi-Fi SSID, MQTT broker 주소 같은 값은 코드 상단이나 별도 헤더에 모아서 관리한다.

### 6-4. 코드 공유 시 포함해야 할 정보

커밋 또는 팀 공유 메시지에는 아래를 포함한다.

- 무엇을 구현했는지
- 어떤 보드에서 테스트했는지
- 무엇이 아직 미완성인지
- 통합 시 다른 팀원이 알아야 할 점이 무엇인지

---

## 7. 통합 전에 하는 체크

통합 전 회의에서 아래를 하나씩 확인한다.

1. 각 노드의 핀맵이 문서화되어 있는가
2. 각 노드의 빌드가 독립적으로 성공하는가
3. 각 노드가 UART 로그를 정상 출력하는가
4. MQTT broker 주소와 포트가 모두 동일한가
5. 토픽 이름과 payload 형식이 모두 같은가
6. 모드와 명령 문자열이 정확히 일치하는가
7. 센서 단위와 값 범위가 문서와 같은가
8. 명령 수신 실패 시 재시도 또는 무시 정책이 정해졌는가
9. Wi-Fi가 끊겼을 때 로그가 남는가
10. 발표용 시연 순서가 정리되었는가

---

## 8. 통합 테스트 순서

통합 테스트는 아래 순서를 추천한다.

1. `노드 A` 단독 실행으로 환경 데이터가 정상 발행되는지 확인
2. `노드 B` 단독 실행으로 테스트 명령에 따라 LED/서보가 동작하는지 확인
3. `노드 C` 단독 실행으로 가짜 센서값을 넣어 명령 생성이 되는지 확인
4. `노드 A + 노드 C` 연결로 센서 수신 및 상태 판단 확인
5. `노드 B + 노드 C` 연결로 명령 송신 및 장치 제어 확인
6. `노드 A + 노드 B + 노드 C` 전체 연결 후 AUTO 모드 시연
7. 전체 연결 상태에서 MANUAL 모드 시연

---

## 9. 문제 발생 시 역할 분리

통합 중 문제가 생기면 아래처럼 구분해서 찾는다.

- 센서값이 이상하다: `노드 A` 문제부터 확인
- LED/서보가 안 움직인다: `노드 B` 문제부터 확인
- 데이터는 오는데 판단이 이상하다: `노드 C` 문제부터 확인
- 아무 것도 안 이어진다: `Wi-Fi/MQTT 설정`부터 확인

한 번에 전체를 의심하지 말고, `센서`, `통신`, `판단`, `출력` 순서로 나눠서 점검한다.

---

## 10. 발표 직전 최종 체크리스트

- 모든 Pico 전원 공급이 안정적인가
- 브레드보드 배선 사진 또는 그림이 있는가
- 노드별 담당자 발표 순서가 정해졌는가
- AUTO 시연과 MANUAL 시연을 모두 준비했는가
- 조도 변화, 온도/습도 변화 시연 방법을 정했는가
- 오류가 나도 설명 가능한 백업 시나리오가 있는가

---

## 11. 추천 진행 순서

처음부터 통신까지 한 번에 하지 말고 아래 순서로 진행한다.

1. 각 노드 하드웨어 단독 제어 성공
2. UART 로그 정리
3. MQTT 연결 성공
4. 고정 메시지 송수신 성공
5. 실제 센서값/제어명령 연결
6. AUTO 모드 완성
7. MANUAL 모드 완성
8. 전체 발표 시나리오 정리

이 순서를 지키면 초보 팀원도 어디까지 완료했는지 명확하게 파악하기 쉽다.
