# 노드 A 개발 문서

## 1. 노드 A 역할

노드 A는 `환경 센서 노드`이다.

주요 역할:

- 조도 측정
- 온도 측정
- 습도 측정
- 측정값 UART 출력
- 중앙 관리 노드가 사용할 수 있도록 MQTT로 환경 데이터 전송
- 주기적 heartbeat 전송

노드 A는 `값을 정확하고 안정적으로 읽는 것`이 가장 중요하다.

---

## 2. 연결 장비

필수 장비:

- `Pico 2 W`
- `조도 센서`
- `온습도 센서`
- `브레드보드`
- `점퍼선`

선택 장비:

- 로컬 테스트용 `버튼`
- 상태 확인용 `LED`

---

## 3. 노드 A 완료 목표

노드 A 담당자는 아래가 되면 1차 완료이다.

1. 전원 인가 후 부팅 로그가 나온다.
2. 조도값이 주기적으로 읽힌다.
3. 온도/습도값이 주기적으로 읽힌다.
4. 측정값이 UART에 보기 좋게 출력된다.
5. 측정값이 합의한 형식으로 `house/env`에 발행된다.
6. Wi-Fi 또는 센서 오류 시 로그가 출력된다.

---

## 4. 구현해야 할 세부 기능

### 4-1. 센서 초기화

- ADC 또는 I2C 센서를 초기화한다.
- 초기화 성공/실패를 UART에 출력한다.
- 실패 시 무한 대기 대신 재시도 또는 오류 상태 로그를 남긴다.

### 4-2. 조도 측정

- 일정 주기로 조도 센서 값을 읽는다.
- 너무 흔들리면 평균값 또는 간단한 필터를 적용한다.
- 결과는 `0~4095` 또는 사용 중인 ADC 스케일에 맞게 문서화한다.

### 4-3. 온습도 측정

- 온도와 습도를 읽는다.
- 실패한 경우 이전값 유지 여부를 정하고 로그를 남긴다.
- 소수점 자리 수는 한 자리 또는 두 자리로 통일한다.

### 4-4. 데이터 패킷 생성

MQTT payload는 아래 형식으로 고정한다.

```txt
light=275,temp=29.1,humidity=72.0
```

키 이름은 절대 바꾸지 않는다.

### 4-5. heartbeat 발행

- `house/heartbeat/nodeA`로 주기적으로 생존 신호를 보낸다.
- payload는 단순하게 `alive` 또는 timestamp 문자열이면 충분하다.

### 4-6. UART 로그 출력

권장 형식:

```txt
[NODE_A] WIFI OK
[NODE_A] SENSOR OK
[NODE_A] light=275 temp=29.1 humidity=72.0
```

## 5. 권장 개발 순서

노드 A는 아래 순서로 개발하면 안정적이다.

1. 조도 센서만 먼저 연결해서 ADC 값 읽기 성공
2. 온습도 센서만 별도로 연결해서 값 읽기 성공
3. 두 센서를 한 프로그램에서 같이 읽기
4. UART 출력 형식 정리
5. Wi-Fi 연결 코드 추가
6. MQTT broker 접속
7. 고정 문자열 발행 테스트
8. 실제 센서값 발행 테스트
9. heartbeat 추가
10. 오류 처리 정리

---

## 6. 로컬 단독 테스트 항목

다른 노드 없이도 아래 테스트가 가능해야 한다.

### 테스트 1. 센서값 변화 확인

- 조도 센서를 손으로 가리면 값이 바뀌는지 확인
- 온도를 손으로 올리거나 주변 온도 변화에 따라 값이 변하는지 확인
- 습도 변화가 느리면 일단 센서 수신 성공 로그만 확인해도 된다

### 테스트 2. UART 출력 확인

- 1초 전후 주기로 값이 계속 출력되는지 확인
- 값이 `0` 또는 비정상 숫자로 고정되지 않는지 확인

### 테스트 3. MQTT 발행 확인

- PC 또는 다른 도구에서 `house/env`를 subscribe 하여 payload 형식을 확인
- 키 이름 오타가 없는지 확인

### 테스트 4. 오류 처리 확인

- 센서를 일부러 분리했을 때 에러 로그가 나오는지 확인
- Wi-Fi가 끊겼을 때 재연결 로그가 보이는지 확인

- topic 이름이 합의안과 완전히 같다
- UART 로그에 `NODE_A` 식별자가 있다
- 센서 실패 로그가 있다
- Wi-Fi 연결 상태 로그가 있다

## 10. 통합 시 주의사항

- 조도값이 너무 출렁이면 노드 C 판단이 계속 바뀔 수 있다
- 너무 자주 publish 하면 통합 로그가 지저분해질 수 있다
- `humidity`를 `humi`로 바꾸면 통합이 깨질 수 있다
- 소수점 자릿수가 너무 길면 파싱이 번거로워질 수 있다
